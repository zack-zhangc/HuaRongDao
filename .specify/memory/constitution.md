# web-test 项目章程

## 核心原则

### I. 代码质量优先
所有变更都必须使触达代码的可读性、可维护性和可诊断性不低于修改前。
提交前必须通过项目定义的格式化、静态检查、类型检查和构建校验。共享
组件、公共接口和核心领域逻辑必须具有明确输入/输出约束、错误处理和必要
文档; 重复逻辑必须在合并前消除, 或记录为限时技术债并附带偿还计划。

理由: 代码质量退化会放大后续迭代成本, 并直接降低交付速度与故障恢复能力。

### II. 测试标准不可协商
每个功能性变更都必须提供与风险匹配的自动化测试。纯逻辑变更必须有单元
测试, 接口或集成边界变更必须有集成或合约测试, 缺陷修复必须附带可复现且
先失败后通过的回归测试。除纯文案、纯样式或仅构建配置且对运行行为无影响
的变更外, 不得以“时间不足”为由省略测试; 任何例外都必须在计划和任务中
记录原因、风险、补救负责人和到期时间。

理由: 没有可执行测试的需求无法稳定演进, 也无法证明修复真正生效。

### III. 用户体验一致性
所有面向用户的变更都必须延续既有的信息架构、命名、交互反馈和视觉语言,
除非规范中记录了经过批准的偏离理由。规范必须定义受影响流程的加载、
空状态、错误、成功和权限受限状态, 并明确响应式、可访问性和跨端一致性
要求。合并前必须验证核心路径在目标设备和断点上行为一致, 不得引入未经
说明的文案、导航或组件模式分叉。

理由: 一致的体验能降低学习成本、支持成本和误操作率, 也是品牌可信度的
一部分。

### IV. 性能要求前置
所有功能规范都必须声明与范围相关的可度量性能预算, 例如 p95 响应时间、
首屏渲染时间、交互延迟、内存占用、包体积或吞吐量。实施方案必须说明如何
测量当前基线、如何验证交付结果, 并在合并前提供可复现的性能证据。任何
已知会突破预算的方案都必须在计划阶段被标记为违规, 并在复杂度跟踪中记录
理由、替代方案和缓解措施。

理由: 性能是用户体验和系统容量的一部分, 不能在功能完成后再补救。

## 质量基线与交付约束

- `spec.md` 必须同时包含功能需求、用户体验一致性要求和性能要求; 成功
  标准必须可测量且与实现细节解耦。
- `plan.md` 必须在章程检查中逐项说明代码质量门控、测试策略、UX 一致性
  验证方案和性能预算验证方案。
- `tasks.md` 必须为每个用户故事包含实现任务、测试任务、UX 验证任务和
  性能验证任务, 使故事能够独立交付与验收。
- 临时例外必须记录在计划的复杂度跟踪或等效豁免记录中, 包含范围、理由、
  风险、批准人和失效日期; 未记录的例外视为不合规。

## 开发流程与审查门控

- 规范阶段: 任何用户故事在进入计划前, 都必须具备可测试验收场景、关键
  状态定义和至少一条可量化成功标准。
- 计划阶段: 章程检查未通过时不得进入任务分解; 所有 `NEEDS
  CLARIFICATION` 必须在实施前解决, 或显式降级为带责任人的 TODO。
- 实施阶段: 开发顺序必须遵循“测试或验证先于完成声明”; 未补齐测试、
  UX 校验或性能证据的故事不得标记完成。
- 审查阶段: 代码审查必须验证质量门控结果、测试覆盖变化、用户可见差异
  截图或说明, 以及性能验证记录。任一项缺失都应阻止合并。
- 发布阶段: Quickstart 或等效验收流程必须覆盖主路径, 并确认不存在阻断级
  缺陷、明显 UX 回归或超预算性能回归。

## 治理

本章程优先于仓库中的其他工作约定。每次修改章程都必须在同一变更中同步
更新受影响的模板、命令提示或运行指导, 并在章程顶部附带同步影响报告。

章程版本遵循语义化版本控制: 原则删除或被重新定义为 MAJOR, 新增原则或
新增强制性门控为 MINOR, 纯澄清或措辞修订为 PATCH。

合规审查是持续性的: `spec.md`、`plan.md`、`tasks.md`、代码审查和发布
验收都必须显式检查本章程。项目维护者可以批准带失效日期的临时豁免, 但
必须记录风险、缓解措施和责任人; 过期豁免自动失效并重新恢复为阻断项。
